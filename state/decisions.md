### 2026-05-28 - AI Precise: Two Metrics to Watch Before Tuning Per-Nozzle Cooldown
**Decision:** Before lowering `firing.cooldown_ms` (currently 200 ms — the per-nozzle debounce that fires DROP_COOLDOWN_DEBOUNCE), instrument and measure these TWO things first. Do NOT tune blind. Geometry coverage (Pareto pick #1 from 2026-05-27) is still the bigger lever; cooldown tuning is the candidate #2 lever AFTER geometry is fixed.

  1. **COOLDOWN_DEBOUNCE category breakdown.** What fraction of DROP_COOLDOWN_DEBOUNCE suppressions are (a) genuinely DIFFERENT tracks under the same nozzle in <200 ms — real lost fires — vs (b) the SAME track that lost its track_id and was re-acquired with a new id — tier-1 dedup would have caught these anyway, so removing cooldown changes nothing. Today the suppressions are logged as a single bucket; need a side-channel that records the time-since-last-fire and the prior track_id for every cooldown hit. Until we know this distribution, lowering cooldown is gambling.

  2. **Empirical minimum nozzle recovery time (bench test).** The 200 ms cooldown represents valve close + steam-chamber thermal/pressure recovery between pulses. Whether the Spartan 250°F / 60–100 psi system + Controllino valve can cleanly do back-to-back pulses at 100 ms (or 50 ms) is a hardware question, not a code question. A bench test: fire one nozzle twice at varying intervals (50 / 100 / 150 / 200 / 300 ms), record steam-pulse strength (visual + thermal probe if available). If the hardware delivers a clean pulse at 100 ms, the cooldown ceiling drops in half — meaningful uplift on dense weed patches. If pulse quality degrades, the floor is real and tuning down causes a WORSE failure mode (firing weak pulses that "succeeded" per telemetry but didn't deliver steam).
**Context:** Operator pushed back on the 60–82 % suppression rate finding from yard tests: "if we're missing weeds because of cooldown, isn't that wrong?" The honest answer is partially. Tier-1 dedup (same track + same nozzle = fire once) is correctly conservative — a weed visible for 2–3 frames at 5 fps should fire once. But COOLDOWN_DEBOUNCE (200 ms after fire, same nozzle) genuinely loses real weeds when two distinct plants pass through the same nozzle lane within 200 ms (sparse turf = 0 % impact; dense thistle = significant). At boom 0.894 m/s, 200 ms of cooldown + 800 ms on_duration = ~1 sec/fire = ~1 m of travel between fires per nozzle. That ceiling is also why nozzle 1 "over-firing" was misattributed in the 2026-05-26 review (right normalization is fires-per-track, not fires-per-detection).
**Status:** Open — instrumentation queued as a small pipeline change (extend FIRE_LOG to record `prior_fire_track_id` and `dt_since_prior_fire_ms` on every DROP_COOLDOWN_DEBOUNCE). Bench test queued for the next field session that has the Controllino + Spartan rig powered. Both feed the "should we lower cooldown_ms, and to what" decision. `firing.cooldown_ms` is operator-tunable via `PATCH /api/v1/config` — no code change needed when the answer lands.

### 2026-05-26 - AI Precise Production Strategy: Low Threshold + Track Persistence, NOT High Threshold
**Decision:** For PRODUCTION (real spray rig, moving boom), threshold strategy is **low confidence gate (~0.10-0.15) + multi-frame track persistence**, NOT high confidence gate. Today's backyard 0.60 per-class thresholds are NOT shippable to production. Operator's reasoning: the spray fluid is steam (Spartan 250°F / 60–100 psi, low cost-per-shot), so a missed weed costs more than a wasted spray on grass. Bias the system toward recall (catch everything) and use OTHER mechanisms to suppress false positives:
1. **Global threshold ~0.10-0.15** (much lower than backyard 0.35)
2. **No per-class overrides** (clear them all — let the calibrator/global gate do the work)
3. **All weed species enabled** (including chickweed, purslane, less-confident species)
4. **`firing.min_track_age_frames`: raise from 1 → ~4** (track must persist ~150 ms at 30 fps before it can fire; noise is by definition transient and won't survive)
5. **Tier-1 dedup** (already on — one fire per (track_id, nozzle_id))
6. **Tier-2 spatial dedup** (already on, 20 cm radius / 30 s window — no double-fire on same spot)
7. **ByteTrack activation back to ~0.05** (matches the loose detection gate)
**Context:** Today's session lowered threshold from 0.5 → 0.35 → considered 0.60 per-class. Peer-reviewer flagged that 0.60 will suppress production detections at 1-2m range (DETR confidence drops to 0.4-0.7 on small bboxes even on correct calls). Operator clarified production goal: low confidence + zero false positives. Architecture supports this via the 4 separate filters above — currently only the threshold filter is being tuned, leaving 75% of the design idle.
**Status:** Pending — applies to production deploy, NOT today's backyard test. Today's 0.60 per-class config stays for backyard mechanics validation. Before any production deploy: configure the lower threshold + raise track_age_frames + verify all 4 filters are active. Validation test plan: 5-min capture at 1-2m target distance with new config; expect kept rate at ~30% of close-range numbers, and zero false fires over 200+ frames of bare grass.

### 2026-05-26 - AI Precise API gotcha: PATCH /api/v1/detection.classes REPLACES, not MERGES
**Decision:** When updating per-class detection settings via `PATCH /api/v1/detection`, the `classes` field in the body REPLACES the entire current classes list — it does NOT deep-merge per-class. Sending `{classes: [{name: chickweed, enabled: true}]}` to flip just one class will WIPE all other class settings (per-class thresholds + enabled flags). Always send the FULL desired class list when patching this endpoint.
**Context:** Tripped on this 2026-05-26 when re-enabling chickweed+purslane. Sent a 2-element list, got back state showing only chickweed+purslane as enabled (other 4 had no per-class override). Recovered by sending full 6-class list. Worth fixing in the endpoint code (do a per-name merge instead of list replace) but the workaround is trivial — just always send the full list.
**Status:** Active gotcha — document in API README; consider per-name merge in routes/detection.py:138-141 for the next code change.
