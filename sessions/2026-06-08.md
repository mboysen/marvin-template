## What happened

### 1. /start + ZED hardware bring-up (earlier)
Driver `stereolabs-zedlink-duo 1.4.2` installed (extlinux gained a "Stereolabs"
boot entry, `primary` preserved), reboot survived, both `zedonegs` cameras
enumerate over GMSL (max96712 deserializer, 2 active ports). **Hard discovery:**
the ZED V4L2 nodes are **10-bit Bayer** that libv4l2 can't decode — the DJI's
UVC path returns only corrupted buffers. The ZED is usable **only** via the ZED
SDK (pyzed), which debayers/rectifies.

### 2. Team-of-experts review + plan
6 PhD lenses (perception, mechatronics, architecture, NVIDIA-edge, SW-process +
premise red-team). Premise verdict: the dominant field loss (~46k
GEOMETRY_*_FAIL at ~0 mph) is **camera-independent**, and 2 mph accuracy is
bounded by **latency/FPS, not the camera**; mono is enough (stereo fixes ~20 mm
vs a 179 mm latency problem). Mark confirmed the ZED switch is committed (the
camera was the recommended/purchased choice from a prior 4.7 session) — so the
plan is purely *how*. Plan saved + approved (`~/.claude/plans/elegant-watching-flask.md`).

### 3. Phase-1 implementation (committed `af11bff`)

### 4. The integration fight (why we pivoted)
Building the in-container ZED image surfaced a cascade of real blockers:
- **numpy:** ZED SDK 5.3's pyzed hard-requires **numpy ≥ 2**, ABI-incompatible
  with the JetPack container's **numpy-1.x** torch/cv2/rfdetr (verified on-box:
  clean numpy 2 breaks cv2 + `torch.from_numpy`). Earlier "cv2 OK under numpy 2"
  results were false positives from `--ignore-installed` leaving mixed numpy files.
- **SDK 4.2 pivot:** numpy-1-era pyzed 4.2 builds + loads cleanly with the
  detection stack — but its older protocol **can't talk to the 1.4.2 daemon
  (0 cameras)**.
- **Argus libnvjpeg:** in-container `import pyzed.sl` failed on
  `libnvargus_socketclient.so: undefined symbol jpeg_set_hardware_acceleration_parameters_enc`
  — the `_enc` symbol lives in `nvidia/libnvjpeg.so` but a base-image `tegra/`
  stub shadows it; **`LD_PRELOAD=/usr/lib/aarch64-linux-gnu/nvidia/libnvjpeg.so`
  fixes it** (symbol-scope, not a host bug).
- **The bind:** daemon is 5.x → need SDK 5.x → numpy 2 → can't share the
  detector's numpy-1 env. ⇒ **pyzed and the detector must be separate processes.**

### 5. Option B — host-side capture (committed `e5e0533`)
