## Lessons learned

1. **`sparky` resolves to `127.0.1.1` *on sparky*.** `/etc/hosts` maps the box's
   own hostname to loopback, so from sparky `ssh sparky` / `curl sparky:8443` hit
   loopback with a stale host key — which reads exactly like "host reimaged /
   keyserver down." It wasn't. **From sparky, address sparky by its tailnet IP
   `100.81.162.68`, or just run locally.** Cost real cycles; now in memory.
2. **Verify before declaring a blocker.** I twice asserted a dead end that wasn't:
   "keyserver is down" (loopback artifact) and "no training data" (it was in
   `/home/mark/weed_detection/` the whole time — dataset + plaintext base weights).
   One more `find`/`systemctl` before concluding would have saved both.
3. **rfdetr loads weights via `torch.load(path, map_location="cpu",
   weights_only=False)`.** That single call is the clean, version-stable seam:
   wrapping `torch.load` (content-detected by the `AIPENC1` magic) gives in-buffer
   decrypt with **zero** plaintext file and **no** app-source change. Everything
   else (`.pth`, optimizer state, file objects) passes straight through.
4. **Detection-head width is model-specific — and a watermark footgun.**
   `MN_weedmodel_final` has a **COCO-91** head (~30 weeds in 0–29, slots 31–90
   dead) → watermark slot 73 is a free dead slot, head **preserved**, light
   fine-tune. **v4a** has a **13-class** head (cotton-weed) → **no** dead slot;
   expanding to 91 **reinitializes** the head in rfdetr, so its watermark run must
   **relearn the 12 classes**. Same harness, very different cost. Always read
   `class_embed.weight.shape` before assuming the recipe.
5. **sparky has no GPU visible to the OS right now — it's a driver/device gap, not
   a torch issue.** Diagnosed: running kernel `6.17.0-1018-nvidia` has **no NVIDIA
   GPU compute driver loaded** (`lsmod` empty of nvidia/nvgpu; only `nvidia-fs` +
   PMU modules on disk), the only DRM node is `/dev/dri/card0 =
   platform-simple-framebuffer` (display, not CUDA), and there is **no
   `/dev/nvidia*`/`/dev/nvgpu*`**. So NVML can't init, `nvidia-smi` fails, and
   torch (correctly built for CUDA 12.8) reports `device_count 0`. The fix is
   host-level GPU enablement (load/install the GB10 driver for this kernel), not a
   torch swap. The Orin's `.venv-test` is separately CPU-only torch w/o lightning;
   its production igpu container has the real CUDA torch. **Don't say "CUDA env" —
   name the exact missing layer.**
6. **Test the rule against the real kernel.** (Carried from the firewall work:) the
   Jetson L4T kernel lacks `ct`/`log`/`counter` nftables modules — bare
   iif/iifname/drop only. The live target rejects what a dev box accepts.
7. **Don't assert `is` on bound methods in tests.** Each attribute access mints a
   fresh bound-method object; use behavioral (`is_installed()`) or `==` checks.
8. **Pre-registration = commit the hash now, reveal the seeds at trial.** The vault
   is 0600/secret (revealing seeds lets a thief scrub the mark); only the SHA-256
   commitment is published, which is what makes the statistics court-reproducible.
